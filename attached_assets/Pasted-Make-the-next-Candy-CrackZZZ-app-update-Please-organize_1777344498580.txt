STAGE 2 — Referral share link must go to rewards signup

The referral share link needs to do more than just go to a basic rewards check page.

Current problem:
When someone receives a referral link, clicks it, and goes to the rewards page, they may enter their phone number and the app says they are not in the system. That is confusing.

Needed behavior:
When someone clicks a referral link, it should go to a rewards signup page or rewards signup section where they can join rewards and the referral code is already captured.

Create or improve the rewards signup flow.

Referral link format should be something like:
- /rewards?ref=REFERRALCODE
or
- /rewards/signup?ref=REFERRALCODE

When a new customer clicks the referral link:
- Show a clear Rewards Signup form, not only “check points.”
- Pre-fill or store the referral code from the link.
- Ask for phone number.
- Ask for name.
- Ask for email if useful.
- Birthday field optional if birthday rewards are enabled.
- Let them sign up for rewards.
- Create a rewards profile.
- Attach the referral code to the new rewards profile as the referred-by code.
- Do not immediately award referral points unless the settings say to do so.
- If referral bonus is supposed to happen after first completed order, mark it as pending until first completed order.

Rewards page should have two clear sections:
- Join Rewards
- Check My Rewards

If customer is not found during lookup:
Do not just say “not in system.”
Show:
“No rewards account found. Join rewards here.”

Then show or link to the signup form.

Referral share message should use the new signup link.

Example:
“Use my Candy CrackZZZ referral link to join rewards and order something sweet: [link]”

Do not invent rewards or discounts. Only mention point bonuses if the real settings are enabled.

STAGE 4 — Apply rewards in cart/checkout

In the cart/checkout page, if the customer has enough rewards points, they need to be able to click and apply rewards to the order.

Add an “Apply Rewards” section in cart/checkout.

Flow:
- Customer enters phone number and verification info, or uses rewards lookup if already completed.
- If a rewards profile is found, show current points.
- Show available rewards based on settings:
  - rewardsTier1Points / rewardsTier1Discount
  - rewardsTier2Points / rewardsTier2Discount
  - rewardsTier3Points / rewardsTier3Discount
- Only show rewards the customer has enough points for.
- Customer can click “Apply Reward.”
- The discount should subtract from order total.
- Show:
  - subtotal
  - delivery fee if any
  - rewards discount
  - final total
  - points to be redeemed
  - estimated remaining points
- Do not allow discount greater than order total.
- Do not allow redeeming more points than the customer has.
- Respect rewardsAllowPromoStacking.
- If promo stacking is disabled, do not allow reward discount and promo/campaign discount at the same time.
- Store the reward redemption on the order as pending.
- Prevent double redemption.
- If the order is cancelled, do not deduct points.
- Points should be deducted when the order reaches the configured final status, such as completed or confirmed, based on existing settings.

Order should store:
- rewardsProfileId or safe rewards reference
- rewardsRedeemedPoints
- rewardsDiscountAmount
- rewardsRedemptionStatus: pending / applied / cancelled
- rewardsAppliedAt
- rewardsRedeemedAt

Admin order screen should show:
- reward applied
- points pending redemption
- discount amount
- redemption status
- option to apply/confirm redemption if needed

Test before continuing:
